# BUG-009 packet_proc_core 包头字节内端序与 spec 相反（Byte0 取错 bit 位）

- 日期：2026-07-14
- 版本：0.2.2
- 疑似归属：RTL（packet_proc_core 字节抽取端序），需 orch/rev 定性——见下"分类说明"
- 状态：FIX_READY（2026-07-14 第二次修复，补齐头字段锁存端序——见文末"第二次修复"）
- 最小复现：`make run TEST=ppa_m2_01_test SEED=1`（N-1，最小合法包）；同一根因致 M2-01~M2-07 全部 FAIL

## 现象

M2 单元级 TB 按 spec 附录 A 大端约定装载行为 SRAM：N-1 包 Word0 = `32'h04_01_00_05`
（Byte0=pkt_len=0x04 置于 bit[31:24]，Byte1=pkt_type=0x01 于 bit[23:16]，Byte2=flags=0x00
于 bit[15:8]，Byte3=hdr_chk=0x05 于 bit[7:0]）。DUT 在 DONE 拍输出：

| 字段 | spec 期望（大端，附录 A） | DUT 实测 | DUT 取自 |
| --- | --- | --- | --- |
| res_pkt_len | 4 | 0x05 | mem_rd_data_i[7:0]（=Byte3/hdr_chk） |
| res_pkt_type | 0x01 | 0x00 | mem_rd_data_i[15:8]（=Byte2/flags） |
| type_error | 0 | 1 | 因 pkt_type 读成 0x00（非 one-hot） |
| format_ok | 1 | 0 | 连带 |

即 DUT 把包头四字节的 bit 位映射整体反了：pkt_len 取 bit[7:0]、hdr_chk 取 bit[31:24]，
与 spec 附录 A/B 的"Byte0 在 MSB"恰好相反。

## spec 依据（大端：Byte0 在 bit[31:24]）

- 附录 A（spec.md:826）：`APB_WRITE(0x040, 32'h08_01_00_09); // pkt_len=8, pkt_type=0x01, flags=0, hdr_chk=0x09`
  —— 字面量 `32'h08010009` 的 bit[31:24]=0x08 被注为 pkt_len(Byte0)；且 hdr_chk 注释算式
  `0x08^0x01^0x00=0x09` 仅在 pkt_len=0x08/pkt_type=0x01/flags=0x00 时成立，自洽要求大端。
- B.1（spec.md:866）：`32'h0C_01_00_0D; // pkt_len=12(0x0C)` —— pkt_len 于 MSB。
- B.2（spec.md:874）：`32'h04_03_00_07; // pkt_type=0x03` —— pkt_type 于 bit[23:16]。
- B.3（spec.md:882）：`32'h04_01_00_FF; // hdr_chk=0xFF` —— hdr_chk 于 bit[7:0](LSB)。
- §6.1（spec.md:377）：Word0 = Byte0–3（头部：pkt_len/pkt_type/flags/hdr_chk）；§3.1 字段表 Byte 偏移。

四例一致钉死：Byte0(pkt_len)→bit[31:24]，Byte3(hdr_chk)→bit[7:0]（大端字节序）。

## 期望

DUT 头字段抽取应为 Byte0=mem_rd_data_i[31:24]、Byte1=[23:16]、Byte2=[15:8]、Byte3=[7:0]；
payload 逐字节同理（Byte(4w+0)=[31:24] … Byte(4w+3)=[7:0]），使 sum/xor 按有效字节序累加。

## 定位（rtl/ 端口/波形 debug，仅供根因参考，非期望值来源）

`rtl/packet_proc_core.sv`：
- L75–78 头字段抽取：`pkt_len=[7:0]`、`pkt_type=[15:8]`、`flags=[23:16]`、`hdr_chk=[31:24]`（小端，与 spec 反）。
- L141–144 payload 字节：`byte0_cur=[7:0]` … `byte3_cur=[31:24]`（同样小端）。

## 分类说明（供 orch/rev 路由）

spec 正文 §3.1/§6.1 只规定字节**顺序**（Byte0=pkt_len…），未在正文显式规定字节在 32-bit
word 内的 **bit 位**；字节内端序仅由附录 A/B 数值示例钉死（大端）。故本条严格说是
"RTL 与 spec 附录示例冲突"：
- 若判 spec 附录为准（大端）→ RTL bug，DE 修 L75–78 与 L141–144 的 bit 切片端序；
- 若 rev 反而裁定采用小端（附录示例改写）→ 归 SPEC_CHANGED，附录 A/B/§3.1 需改，且 M1
  已回归的写通路不受影响（M1 未在字节级校验端序）。

DV 参考模型（tb/uvm/core_agent/ppa_core_seq_item.sv predict()/get_word()）按 spec 附录
大端推导，未照抄 RTL；本条不自行改 checker 去迁就 RTL。

## 仲裁结论（rev，2026-07-14）

**方向 (A)：附录 A/B（大端）为准 → 判 RTL bug，归 DE。** 完整仲裁记录见
`doc/evidence/v0.2.3/review-bug-009-arbitration.md`。要点：

- 独立验算纠正一处：hdr_chk 的 XOR 自洽（四头字节整体 XOR=0）对字节置换对称，**大端/小端
  都自洽**，不能用作端序判据。真正钉死大端的是附录注释里显式字段值（pkt_len=8、pkt_type=0x03、
  hdr_chk=0xFF 等只在大端成立）+ 附录 A(err_flag=0、res_pkt_len=8) 与 B.3(隔离 chk_error) 的
  **结果级** 约束（小端会额外触发 length_error/type_error，破坏示例意图）。
- §3.1/§6.1 正文确对 word 内 bit 位留白，但 spec 含附录、附录唯一自洽指向大端且无任何文字支持
  小端；RTL 小端无 spec 锚点，属"RTL 与 spec 附录冲突"而非 spec 双向歧义 → RTL bug。
- M1 packet_sram/apb_slave_if 全程整字搬运、不做字节级 bit 映射，改 M3 切片对 M1 零回归。
- DE 修 `rtl/packet_proc_core.sv` L75-78（头字段：pkt_len=[31:24]…hdr_chk=[7:0]）与
  L141-144（payload：byte0=[31:24]…byte3=[7:0]）；DV(≠修复人)以 `make run TEST=ppa_m2_01_test
  SEED=1` + M2 回归复验关单。DV checker 无需返工。
- 附带（交 arch 走 §8，非本裁决前置）：① 建议在 §3.1/§6.1 补显式 bit 映射澄清注（把附录已隐含
  的大端显式化，非行为变更）；② 附录 B.3(spec.md:881) 注释算式笔误，正确应为 0x04^0x01^0x00=0x05。

## 复验结果（DV，2026-07-14，本地 VM 真跑，关单人≠修复人）

**驳回：commit 9c28fea 修复不完整，状态回退 FIX_READY → OPEN。** 登记的最小复现
`make run TEST=ppa_m2_01_test SEED=1` 复跑仍 **FAIL（UVM_ERROR=9）**；M2 回归 `ppa_m2_01/02/03/06/07`
全 FAIL，仅 `ppa_m2_04/05` PASS（这两条只含单字帧 len=4，见下）。M1 全 9 条 + smoke 全 PASS（无回归）。

### 根因（RTL 复审，仅供 DE 参考，非期望值来源）

9c28fea 已把**组合**头字段抽取（`pkt_len_eff` 等，L76-79）与 payload 字节（`byte0_cur` 等，
L143-146）改为大端 —— 这两处正确。但**遗漏了头字段锁存**（`always_ff` 内 L222-225），仍为小端：

```
pkt_len_q  <= mem_rd_data_i[7:0];    // 应为 [31:24]（Byte0）
pkt_type_q <= mem_rd_data_i[15:8];   // 应为 [23:16]（Byte1）
flags_q    <= mem_rd_data_i[23:16];  // 应为 [15:8] （Byte2）
hdr_chk_q  <= mem_rd_data_i[31:24];  // 应为 [7:0]  （Byte3）
```

结果两套头字段表示不一致：组合 `*_eff`（大端）vs 锁存 `*_q`（小端）。

- `res_pkt_len_o <= pkt_len_eff[5:0]` 等在 PROCESS→DONE 拍写入（L249-257），该拍条件为
  `word_cnt_q == last_word_cnt`；而 `pkt_len_eff = (word_cnt_q==0) ? [31:24] : pkt_len_q`。
- **单字帧**（pkt_len≤4，last_word_cnt=0）：DONE 转换拍 `word_cnt_q==0`，`*_eff` 取大端组合值 → 正确
  → N-1、M2-04/05 PASS。
- **多字帧**（pkt_len≥5，last_word_cnt≥1）：DONE 转换拍 `word_cnt_q≠0`，`*_eff` 退回锁存 `*_q`（小端）
  → `res_pkt_len`=hdr_chk 字节、`res_pkt_type`=flags 字节，三类错误亦据错误字节判定 → N-2/N-3 及
  M2-02/03/06/07 全 FAIL。实测逐条与此机制吻合（如 len=8→res_pkt_len=0x0a=chk；len=33→0x20=chk[5:0]；
  len=100→0x25=chk[5:0]）。

### 建议修复

将 L222-225 锁存改为大端，与 L76-79 组合 `*_eff` 一致：
`pkt_len_q<=[31:24]`、`pkt_type_q<=[23:16]`、`flags_q<=[15:8]`、`hdr_chk_q<=[7:0]`。
（payload L143-146 与组合头 L76-79 已正确，无需再动。）修复后 DV 以同一 TEST+SEED + M2 回归复验关单。

## 第二次修复（DE，2026-07-14，与首次修复人不同实例）

### 根因（补充）

首次修复 9c28fea 只覆盖两条**组合抽取**路径（头字段 `*_eff` L76-79、payload `byte*_cur`
L143-146，均已正确大端），遗漏第三处：`always_ff` 内**头字段锁存**（L221-226 原 L222-225）仍
小端。多字帧（`last_word_cnt≥1`）在 PROCESS→DONE 拍 `word_cnt_q≠0`，`pkt_len_eff` 等选择器退回
锁存 `*_q`（小端）→ `res_pkt_len`/`res_pkt_type` 等取错字节。单字帧 DONE 拍 `word_cnt_q==0` 走
组合大端值，故 N-1/M2-04/05 早已 PASS，恰好掩盖了锁存路径的端序缺陷。

### 本次修改

`rtl/packet_proc_core.sv` 头字段锁存四行改为大端，与组合 `*_eff` 端序完全一致：

```
pkt_len_q  <= mem_rd_data_i[31:24];  // Byte0
pkt_type_q <= mem_rd_data_i[23:16];  // Byte1
flags_q    <= mem_rd_data_i[15:8];   // Byte2（保留字段，提取但不校验）
hdr_chk_q  <= mem_rd_data_i[7:0];    // Byte3
```

仅改端序 bit 切片，不动 FSM/时序结构/判定逻辑，无新行为。

### payload 路径独立核实（未预设"只有头字段有问题"）

独立通读确认 payload **不存在**头字段那样的"组合 + 锁存"两套镜像：`byte0_cur..byte3_cur`
（组合，L143-146，已大端）直接馈入 `sum_next`/`xor_next`（L148-158），后者每 PROCESS 拍**增量
累加**进 `sum_acc_q`/`xor_acc_q`（L230-231）。累加器存的是**已按大端解析后的算术和/XOR 中间值**，
不是原始字节的锁存镜像，因此无任何小端残留可修。DONE 拍 `res_payload_*` 取 `sum_next`/`xor_next`
（L252-253），同源大端。结论：payload 路径正确，无第三处端序遗漏；本缺陷唯一残留点即头字段锁存。
